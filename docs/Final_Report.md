# 1. Manual Reconnaissance

Before running Cartographer, I performed a **manual exploration of the repository** to simulate a realistic Day-One workflow for a Forward Deployed Engineer (FDE) encountering an unfamiliar system.

The goal of this reconnaissance was to answer five typical onboarding questions:

1. Where does data enter the system?
2. What are the most critical output datasets?
3. What would break if a module changes?
4. Where is business logic implemented?
5. Which files change most frequently?

This manual analysis serves as a **ground truth baseline** for evaluating Cartographer’s automated analysis.

---

## Primary Data Ingestion Path

Initial exploration showed that raw datasets are transformed through **staging models** located in:

```
models/staging/
```

Key staging models include:

* `stg_customers.sql`
* `stg_orders.sql`
* `stg_payments.sql`

These models normalize raw data before it flows into downstream analytics models.

The staging layer therefore acts as the **primary ingestion boundary**, separating raw source systems from curated analytical tables.

---

## Critical Output Datasets

Manual inspection suggests that the main output datasets are:

* `customers`
* `orders`
* `payments`

These datasets are generated by SQL transformations referencing staging models through **DBT `ref()` dependencies**.

This pattern forms a **directed acyclic graph (DAG)** of transformations.

---

## Blast Radius (Change Impact)

Tracing the dependency graph manually shows that modifying:

```
models/staging/stg_customers.sql
```

would propagate changes to several downstream models.

This indicates that staging models represent **high-impact nodes** in the transformation DAG.

Understanding these dependencies is critical when modifying transformation logic.

---

## Business Logic Location

Most transformation logic resides inside:

```
models/
```

The repository primarily uses **SQL-based transformation logic rather than Python orchestration**, which is typical for DBT-based data pipelines.

---

## Git Velocity Observations

Based on repository history, frequently modified files include:

* `stg_orders.sql`
* `stg_payments.sql`
* `schema.yml`

These files likely represent areas where business logic or schema definitions evolve frequently.

---

## Difficulties Encountered During Manual Exploration

Several challenges emerged during manual exploration:

* No centralized architectural documentation
* Difficulty visualizing model dependencies
* Need to manually trace `ref()` relationships across SQL models

These difficulties arise from **structural properties of DBT repositories**. Dependency relationships are embedded inside SQL files using the `ref()` abstraction, meaning they are distributed across many files rather than defined in a single configuration.

As a result, understanding the transformation pipeline requires **manually opening multiple files and reconstructing the dependency graph mentally**.

This observation directly informed the design priorities of Cartographer:

* the **Hydrologist agent** was designed to automatically reconstruct lineage graphs from SQL models
* the **Surveyor agent** builds structural dependency graphs
* the **Navigator agent** provides interactive querying to explore these relationships

By automating dependency reconstruction and documentation generation, Cartographer aims to eliminate the most time-consuming aspects of manual repository reconnaissance.

# 2. System Architecture

Cartographer uses a **multi-agent pipeline architecture** to analyze repositories and generate structured knowledge artifacts.

The system consists of five primary agents:

* **Surveyor** — static code analysis
* **Hydrologist** — data lineage reconstruction
* **Semanticist** — semantic interpretation
* **Archivist** — artifact generation
* **Navigator** — graph-based query interface

## Design Tradeoffs

Several architectural design decisions influenced the system implementation.

### NetworkX vs Graph Database

Cartographer uses **NetworkX rather than a full graph database (e.g., Neo4j)**.

This decision was made because:

* repository graphs are typically **small to medium sized**
* analysis runs **offline as a batch pipeline**
* NetworkX integrates easily with Python agent workflows

Using a graph database would enable persistent storage and distributed querying, but would introduce unnecessary operational complexity for the current use case.

---

### Static Analysis vs Runtime Instrumentation

Cartographer relies primarily on **static analysis** rather than runtime instrumentation.

Advantages:

* works on any repository without executing code
* fast and reproducible

Limitations:

* cannot observe runtime behavior
* struggles with dynamically generated dependencies

# 3. Accuracy Analysis

Cartographer’s automated analysis was compared with the manual reconnaissance results.

---

## Correct Observations

The system successfully identified several important structural properties:

* staging models as the primary ingestion layer
* dataset lineage relationships derived from SQL parsing
* downstream dependencies between staging and final models
* blast radius effects when modifying staging tables

The Hydrologist-generated lineage graph closely matches the **DBT model DAG**, indicating that SQL parsing correctly reconstructed transformation dependencies.

---

## Partially Correct Observations

The Semanticist agent produced generally correct summaries of module purpose but sometimes lacked domain-specific detail.

For example, SQL models performing aggregations were summarized as **generic data processing logic** rather than explicitly identifying the transformation semantics.

This occurs because Semanticist relies on **embedding-based summarization rather than symbolic program reasoning**.

---

## Incorrect or Missing Observations

Some system outputs were inaccurate or incomplete.

| Issue                  | Responsible Component         |
| ---------------------- | ----------------------------- |
| Dynamic SQL references | Hydrologist (SQLGlot parsing) |
| Macro expansion        | Hydrologist                   |
| Generic summaries      | Semanticist                   |
| Hub misclassification  | Surveyor (centrality metrics) |

Hydrologist uses static SQL parsing through SQLGlot, which cannot fully interpret dynamic query generation or DBT macro expansion.

Similarly, Surveyor relies on **graph centrality metrics** to identify important modules, which may incorrectly prioritize frequently referenced utility modules.

# 4. Limitations and Failure Modes

Cartographer currently has several limitations.

These can be divided into **engineering gaps** and **fundamental static-analysis constraints**.

---

## Engineering Limitations (Potentially Fixable)

### Macro Expansion Handling

DBT macros can dynamically generate SQL queries.

The current SQL parsing approach does not expand macros, which may lead to incomplete lineage graphs.

---

### Improved Semantic Interpretation

The Semanticist relies on embeddings and LLM summarization.

More advanced semantic analysis could incorporate:

* SQL structure analysis
* transformation pattern detection
* schema reasoning

---

## Fundamental Static Analysis Constraints

Some limitations arise from the nature of static code analysis.

### Runtime Behavior Visibility

Static analysis cannot observe runtime behavior such as:

* dynamically generated queries
* conditional pipeline execution
* external service interactions

---

### Architectural Intent vs Structural Metrics

Graph metrics such as PageRank identify highly connected modules but cannot infer **architectural importance**.

A module referenced frequently may be structurally central but not conceptually important.

---

## Confidently Incorrect Output Scenario

One potential failure scenario involves **dynamically generated dataset names**.

If a SQL macro constructs table names dynamically, the Hydrologist may fail to detect upstream dependencies.

In this case, the generated lineage graph may incorrectly show a dataset as having **no upstream sources**, even though dependencies exist at runtime.

Because the system currently lacks uncertainty detection, this failure could produce **confidently incorrect lineage output**.

# 5. FDE Deployment Applicability

Cartographer is designed to support **Day-Zero reconnaissance during Forward Deployed Engineering engagements**.

---

## Cold Start Workflow

An FDE onboarding onto an unfamiliar system could use Cartographer as follows:

1. Clone the client repository
2. Run:

```
uv run python -m src.cli data/repo-name
```

3. Cartographer generates:

* module dependency graphs
* dataset lineage graphs
* CODEBASE.md documentation
* onboarding briefing artifacts

Within minutes, the engineer receives a structured overview of the repository architecture.

---

## Interactive Exploration with Navigator

After initial analysis, the **Navigator agent** supports interactive repository exploration.

Example queries include:

```
navigator.query("upstream sources for customers dataset")

navigator.blast_radius("stg_orders.sql")

navigator.query("purpose of models/orders.sql")
```

This enables engineers to investigate system behavior without manually searching through code.

---

## Living Documentation

Cartographer generates:

```
CODEBASE.md
```

This file functions as a **living architectural document**.

Each time Cartographer is re-run, the documentation is regenerated and can be committed alongside the repository.

This ensures the documentation evolves alongside the codebase.

---

## Remaining Human Responsibilities

Despite automation, the FDE still performs important validation tasks:

* confirming lineage accuracy for dynamic SQL patterns
* validating semantic summaries
* verifying architectural assumptions with system owners

Cartographer therefore acts as a **force multiplier**, accelerating understanding rather than fully replacing manual investigation.

# 6. Self-Audit: Cartographer on My Week-1 Repository

To evaluate performance on a Python-heavy system, I ran Cartographer on my **Week-1 AI-Native IDE repository**.

---

## Observations

Cartographer successfully detected:

* module dependencies between orchestration components
* middleware layers controlling agent behavior
* architectural hubs in orchestrator modules

These outputs demonstrate that the system performs particularly well on **Python-based repositories**, where AST analysis captures structural relationships accurately.

---

## Discrepancies

Two inaccuracies appeared.

### Hub Overestimation

Graph centrality occasionally ranked utility modules as major hubs.

This occurs because PageRank measures **reference frequency rather than architectural importance**.

---

### Generic Semantic Summaries

Semanticist sometimes produced overly general summaries that did not capture orchestration responsibilities.

This limitation reflects the current reliance on **embedding-based summarization rather than deeper program reasoning**.

# Conclusion

Brownfield Cartographer demonstrates how a **multi-agent repository intelligence system** can automate the initial stages of developer onboarding.

By combining:

* structural code analysis (Surveyor)
* data lineage reconstruction (Hydrologist)
* semantic interpretation (Semanticist)
* artifact generation (Archivist)
* interactive graph querying (Navigator)

the system transforms unfamiliar repositories into structured architectural knowledge.

Although limitations remain in dynamic SQL analysis and semantic interpretation, Cartographer significantly reduces the time required to understand complex codebases and provides a foundation for AI-assisted software exploration.
